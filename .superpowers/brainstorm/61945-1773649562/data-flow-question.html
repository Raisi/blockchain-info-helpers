Datenfluss: Wie realistisch soll es sein?

Tabs 1–3 rechnen über reellen Zahlen (glatte Kurve). Tab 4 nutzt echtes secp256k1 über einem endlichen Körper (mod p). Diese Welten sind mathematisch verschieden — ein Punkt von Tab 3 kann nicht direkt zum Public Key in Tab 4 werden.

Das Kernproblem

Tab 1–3: y² = x³ + 7 über ℝ (reelle Zahlen, glatte Kurve)
Tab 4: y² = x³ + 7 über 𝔽ₚ (endlicher Körper, diskrete Punkte)

Gleiche Gleichung, gleiche Operationen — aber andere "Welt". Der Datenfluss muss diese Brücke konzeptionell schlagen.

A

Konzeptioneller Fluss (empfohlen)

Die Pipeline zeigt den gleichen Prozess, nicht die gleichen Zahlen. Jeder Tab demonstriert einen Baustein, und die Pipeline-Bar verbindet sie narrativ:

  • Tab 1→2: "Du siehst die Kurve. Jetzt lernst du, wie man Punkte darauf addiert."
  • Tab 2→3: "Addition wiederholt = Skalarmultiplikation. Das ist die Einwegfunktion."
  • Tab 3→4: "Genau so funktioniert es bei Bitcoin — nur über einem endlichen Körper. Hier siehst du es mit echten Zahlen."

Tab 4 zeigt dann zusätzlich: "Der Skalar n, den du gerade in Tab 3 gewählt hast, wäre dein Private Key. Hier ist, wie das mit echtem secp256k1 aussieht:" → und generiert einen echten Key mit dem gleichen Konzept.

B

Hybrider Fluss

Tab 1–3 teilen echte Punkte untereinander (reelle Kurve). Beim Übergang zu Tab 4 gibt es einen expliziten "Weltenwechsel":

  • Tab 1→2→3: Punkt P aus Tab 2 wird Basispunkt in Tab 3, Skalar n wird übernommen
  • Tab 3→4: Animierter Übergang: "Gleiche Mathe, anderes Zahlenfeld" — der reelle Punkt wird symbolisch zum secp256k1-Generator G, der Skalar n wird zum Private Key

Der User sieht: "Mein Skalar 7 aus Tab 3 → hier ist 7 × G mit echtem secp256k1"
